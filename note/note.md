### 1 项目描述

* `tcmalloc`是Google开源的一个高性能内存池,相较于`glibc malloc`,`tcmalloc`在大内存的申请上性能会更高

* 关于池化技术,简单来说就是对于频繁向系统申请的资源,申请本身需要一定的性能开销,所以不如一开始就申请很多资源,并将其放进一个池子,等到要用的时候直接拿去用就行

* 那么字面意思,意味着我们申请的资源是内存

#### 2 `brk`与`mmap`

##### 2.1 作用与机制
* 这两个接口都是系统调用,他们的定位都是申请一定空间
* 区别在于,`brk`被用于申请堆空间,而`mmap`被用于申请在文件映射区域的空闲处的空间
* 两者都是申请虚拟内存,第一次向申请的内存写入时会陷入内核态触发缺页中断
* 更深入地说,我们知道,一个进程对于空间的划分是通过`vm_area_struct`(称`VMA`)实现的
```C
// 内核源代码
struct vm_area_struct {

    struct mm_struct * vm_mm;   /* The address space we belong to. */
    unsigned long vm_start;     /* Our start address within vm_mm. */
    unsigned long vm_end;       /* The first byte after our end address
                       within vm_mm. */

    /* linked list of VM areas per task, sorted by address */
    struct vm_area_struct *vm_next;

    pgprot_t vm_page_prot;      /* Access permissions of this VMA. */
    unsigned long vm_flags;     /* Flags, listed below. */

    struct rb_node vm_rb;
    //......

}
```
* 对于一个已经加载进内存的可执行文件,堆区的`VMA`是已经被创建好的,本质上`brk`只会做调整堆区结束位置的行为
* 但对于`mmap`,`mmap`会在空闲位置直接开一个新的`VMA`,劣势在于,开一个新的`VMA`,性能消耗比较大,因为开`VMA`不仅其自身这个大结构体需要创建,还需要将自己插入一个红黑树,同时每个`VMA`都是一个链表中的节点,那么我们还需要将新的`VMA`插入这个链表中
* 可以说,`brk`其实是针对`heap VMA`做特殊优化的接口

* `glibc malloc`的底层也是`brk`和`mmap`,他会在申请空间小于`128K`的时候使用`brk`开辟空间,大于`128K`的时候会使用`mmap`开辟空间
* 关于`glibc free`的释放机制,释放开辟在堆区的空间时,会先释放其资源,让该空间保持空闲但不交还给操作系统,然后做一个判断,如果堆区末尾存在一片连续的空闲空间且大小大于`128K`,那么则会释放该空间,否则就不做释放,这意味很容易产生内存碎片,而释放开辟在文件映射区域的空间时,一般可以直接释放,几乎不存在什么"空闲空间"的情况,不过如果做部分空间释放的话,性能方面会产生一定问题

* 关于释放开辟在文件映射区域的空间的性能问题,如果只是删除一整个`VMA`还好,但现实是存在一种"部分释放"的情况,比方说开辟了`256MB`的空间,但过程中我们需要释放最中间的`128MB`,那此时就出现问题了,你会发现此时如果释放的话这连续的`256MB`就会被拆分成两个不连续的`64MB`,实际情况下,这新的两个`64MB`的空间会单独构成一个`VMA`,意味着释放空间可能不会导致`VMA`减少,可能还会导致`VMA`增多,甚至要根据情况做红黑树调整,过于麻烦了,效率低下

* 综上我们使用这两个接口时,一定要注意:
	1. 频繁申请小空间,使用`brk`效率更高,且注意内存碎片问题
	2. 申请大空间时,使用`mmap`,尽可能少的申请,或者一次性申请很多空间,尽最大可能不做拆分一块连续空间的操作

* 另外,释放堆区的接口仍然是`brk`,释放文件映射区的接口叫做`munmap`,具体原因我们看下一小节就知道了
